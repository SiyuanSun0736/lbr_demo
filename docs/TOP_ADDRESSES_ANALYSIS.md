# Top地址统计与栈展开功能

## 功能概述

在程序退出前，自动统计LBR记录中访问最多的前5个地址，并对这些热点地址进行详细的栈展开分析。

## 实现原理

### 1. 数据收集阶段
程序运行期间，会统计所有LBR分支记录中出现的地址（包括From和To地址），并记录：
- 地址本身
- 访问次数
- 最后一次采样时的寄存器状态（RIP、RSP、RBP）

### 2. 分析阶段
当程序收到退出信号（Ctrl+C）时：
1. 对所有收集的地址按访问次数排序
2. 选择访问最多的前5个地址
3. 解析每个地址的符号信息
4. 使用记录的寄存器状态进行精确的栈展开

## 使用方法

### 基本用法
```bash
sudo ./lbr-demo -pid <目标PID>
# 运行一段时间后，按 Ctrl+C 退出
# 程序会自动显示Top 5地址的分析结果
```

### 命令行参数
- `-unwind-on-exit`: 是否在退出时进行Top地址分析（默认：true）
- `-unwind`: 是否启用栈展开功能（默认：true）
- `-sframe`: 是否使用SFrame进行符号解析（默认：true）

### 禁用退出分析
```bash
sudo ./lbr-demo -pid <PID> -unwind-on-exit=false
```

## 优势

### 1. 性能优化
- **避免实时开销**：不在每次采样时都进行栈展开，减少运行时开销
- **集中分析**：只分析最重要的热点地址

### 2. 准确性提升
- **保存寄存器状态**：eBPF程序在采样时记录RIP、RSP、RBP
- **时间一致性**：栈展开使用采样时刻的寄存器状态，而非退出时的状态
- **避免状态变化**：不受进程继续运行导致的状态变化影响

### 3. 分析价值
- **热点识别**：快速定位程序中执行最频繁的代码路径
- **性能优化指导**：了解热点函数的调用链
- **问题诊断**：识别异常执行路径

## 技术细节

### 数据结构
```go
type AddrContext struct {
    Addr  uint64  // 地址
    Rip   uint64  // 采样时的指令指针
    Rsp   uint64  // 采样时的栈指针
    Rbp   uint64  // 采样时的帧指针
    Count uint64  // 访问次数
}
```

### eBPF增强
修改后的BPF程序在记录LBR时同时保存：
```c
struct lbr_data {
    __u64 pid_tgid;
    __s64 nr_bytes;
    __u64 rip;  // 程序计数器
    __u64 rsp;  // 栈指针
    __u64 rbp;  // 帧指针
    struct perf_branch_entry entries[MAX_LBR_ENTRIES];
};
```

### 栈展开方法
```go
// 使用保存的寄存器状态创建上下文
ctx := lbr.NewUnwindContextFromRegs(stat.Rip, stat.Rsp, stat.Rbp)

// 从该上下文开始栈展开
frames, err := sframeResolver.UnwindStackFromContext(ctx, 16)
```

## 注意事项

1. **寄存器状态的有效性**：如果eBPF未能记录寄存器状态（Rip=0），则无法进行栈展开

2. **内存访问**：栈展开需要读取进程内存，确保进程在分析时仍在运行

3. **符号信息**：需要目标程序包含调试符号或SFrame数据才能获得详细的函数名和行号

4. **统计准确性**：地址访问统计基于采样数据，不是精确计数

## 与实时栈展开的对比

| 特性 | 实时栈展开 | 退出时Top地址分析 |
|------|-----------|------------------|
| 性能开销 | 高 | 低 |
| 分析范围 | 所有采样 | Top 5地址 |
| 准确性 | 依赖当前状态 | 使用采样时状态 |
| 适用场景 | 详细追踪 | 热点分析 |

## 组合使用建议

- **开发调试阶段**：启用实时栈展开（`-unwind=true`）
- **性能分析阶段**：仅启用退出分析（`-unwind=false -unwind-on-exit=true`）
- **生产环境监控**：禁用所有栈展开（`-unwind=false -unwind-on-exit=false`）
