

**能处理，但有前提条件。** 代码中已经实现了完整的共享库支持，但实际效果取决于库本身是否携带 `.sframe` 节。

**已实现的能力：**

1. **地址映射**：`loadLibraryMappings()` 在初始化时从 `/proc/pid/maps` 加载所有 `.so` 映射，记录运行时基址，正确处理 ASLR。

2. **SFrame 展开路径**（sframe_resolver.go）：
   ```
   if ctx.PC in [baseAddr, baseAddrEnd):  // 主程序
       sframeData = r.sframeData
   else:
       // 在 r.mappings 中按 PC 范围查找对应共享库的 SFrame 数据
   ```

3. **符号解析**：共享库的动态符号表（`DynamicSymbols()`）在 `loadLibraryMapping()` 中加载，绝大多数公开函数都能解析到名称。

4. **回退机制**（`doUnwindStackWithSFrame`）：若某帧所在库无 `.sframe` 数据，自动回退 FP-based 展开，直到进入有 SFrame 数据的帧为止。

**实际限制：**

| 情况 | 结果 |
|------|------|
| 共享库含 `.sframe` 节（用 GCC ≥ 14 + `-fasynchronous-unwind-tables` 编译） | 完整 SFrame 展开 |
| 共享库无 `.sframe`（如系统 libc 发行版二进制） | 回退 FP 展开，需库以 `-fno-omit-frame-pointer` 编译 |
| FP 也不可用（如 `-O2` 默认省略帧指针） | 该帧处栈展开中断 |
| `phase1SFrameResolver` 在 Phase 1 期间由目标进程创建，Phase 2 中进程若已退出 | 共享库映射路径仍有效，但 `/proc/pid/mem` 读栈失败（已有 BPF 快照兜底） |

**实际上 libc 等系统库大概率走 FP 回退路径**，能否成功展开取决于目标进程的编译选项。
